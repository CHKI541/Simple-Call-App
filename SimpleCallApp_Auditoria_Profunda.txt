1. Auditoría del Sistema de Login
El flujo de autenticación tiene 2 bugs adicionales no detectados anteriormente:
✗ BUG 1 — reloadUser() sin await real — puede no sincronizar
Problem
a:
En AuthViewModel.login(), la llamada es: repo.reloadUser() sin capturar su resultado. reloadUser() es una
suspend function que puede tardar 200-500ms en red lenta. Si la red es lenta, isEmailVerified puede leerse
ANTES de que reload() complete. El usuario ve 'Email no verificado' aunque ya lo verificó.
Solución
:
Esperar explícitamente: repo.reloadUser() sin ignorar el resultado. Adicionalmente, agregar un reintento si falla
la sincronización.
■ BUG 2 — Usuario Google va a ChooseNumber aunque ya tiene número
Problem
a:
PermissionActivity llama userRepo.getUser(uid) para decidir si ir a ChooseNumber o MainActivity. Si el usuario
inicia sesión con Google por primera vez Y ya tenía cuenta (email vinculado), el documento en Firestore existe
pero puede tardar en cargarse. Si getUser() devuelve null por timeout de red, va a ChooseNumber y le pide
número de nuevo.
Solución
:
Agregar retry con delay en getUser(). Si falla 3 veces, ir a MainActivity directamente (el número se puede elegir
más tarde desde el perfil).

2. Auditoría del Sistema de Llamadas
Este es el área más crítica. Se encontraron 5 bugs que afectan directamente que la llamada suene, se conecte,
tenga buen audio y se pueda cortar.
✗ BUG 3 — CRÍTICO: FCM no incluye el SDP offer — receptor no puede contestar
Problem
a:
MyFirebaseMessagingService.handleCall() recibe el callId y callerUid pero el 'offer' SDP viene en data['offer']. El
problema real: la app NUNCA ENVÍA la notificación FCM. La guía muestra cómo RECIBIR la notificación pero
nunca explica cómo ENVIARLA. Sin un servidor (Firebase Cloud Functions o backend propio) que envíe el FCM
push, el receptor NUNCA recibe la llamada si la app está cerrada. El ForegroundService y la señalización
Firestore solo funcionan si la app está abierta.
Solución
:
Para que las llamadas lleguen con la app cerrada se necesita Firebase Cloud Functions. Mientras tanto: la app
debe escuchar llamadas entrantes via Firestore listener en un Service permanente. Ver código completo en
sección 2.1.
✗ BUG 4 — ForegroundService arranca antes del offer — ringtone sin llamada real
Problem
a:
En MyFirebaseMessagingService.handleCall(), se llama startForegroundService() SIEMPRE que llega una
notificación FCM. Si la notificación llega pero el offer SDP en Firestore no está disponible aún (race condition de
red), el ringtone suena pero createAnswer() falla porque remoteSdp está vacío.
Solución
:
Verificar que el offer SDP es válido antes de iniciar el ForegroundService. El offer ya llega en el FCM data
payload — validarlo antes de proceder.
✗ BUG 5 — Sin AudioFocus — audio del sistema interfiere con la llamada
Problem
a:
WebRTC maneja su propio audio PERO Android requiere que la app solicite AudioFocus para interrumpir
música, podcasts u otras apps de audio. Sin esto: si el usuario tiene música sonando, ambos audios se mezclan
y se escucha horrible. También puede causar que el auricular y el altavoz no respondan correctamente.
Solución
:
Solicitar AudioFocus en CallActivity.setupAudio() y liberarlo en hangUp(). Ver código en sección 2.2.
✗ BUG 6 — onTrack() accede a rtc que puede ser null
Problem
a:
En CallActivity, el observer de PeerConnection se crea con una lambda que captura 'rtc'. Si onTrack() se llama
desde un hilo de WebRTC DESPUÉS de que hangUp() llamó rtc.disconnect(), rtc.setRemoteAudioTrack() lanza
NullPointerException. Esto puede ocurrir en cierres de llamada rápidos o al recibir tracks tardíos.
Solución
:
Verificar que rtc no es null antes de llamar setRemoteAudioTrack(). Usar 'rtc?.setRemoteAudioTrack(it)' con
safe call.
■ BUG 7 — setRemoteAnswer() desde hilo Firestore — race condition WebRTC
Problem
a:
listenForAnswer() dispara el callback desde el hilo principal de Firestore listeners. setRemoteAnswer() →
peerConnection.setRemoteDescription() debe llamarse desde un hilo específico en algunas versiones de
WebRTC. En Android, PeerConnection es thread-safe para la mayoría de operaciones, pero
setRemoteDescription después de createOffer puede crear una carrera si ICE candidates llegan antes de que
setRemoteDescription complete.
Solución
:
Ejecutar setRemoteAnswer() en el hilo principal con runOnUiThread o en un Dispatcher.Main. Esto garantiza el
orden de operaciones correcto.
2.1 — Solución para llamadas con app cerrada: IncomingCallService
Sin Cloud Functions, la única forma de recibir llamadas con la app cerrada es un Foreground Service
permanente que escuche Firestore. Este Service se inicia cuando el usuario hace login y escucha llamadas
entrantes.
i OPCIÓN RECOMENDADA A CORTO PLAZO: El receptor debe tener la app abierta para recibir llamadas. Esto
es lo mismo que Telegram si no tiene notificaciones configuradas. Funciona perfectamente en pruebas con 2
teléfonos donde ambos tienen la app abierta.
service/IncomingCallListenerService.kt — ARCHIVO NUEVO
package com.simplecallapp.service
import android.app.*; import android.content.*; import android.os.*
import androidx.core.app.NotificationCompat
import com.google.firebase.auth.FirebaseAuth
import com.google.firebase.firestore.FirebaseFirestore
import com.google.firebase.firestore.ListenerRegistration
import com.simplecallapp.ui.call.CallActivity
// Este service escucha llamadas entrantes aunque la app esté en segundo plano.
// Iniciar en MainActivity.onCreate() si el usuario está logueado.
class IncomingCallListenerService : Service() {
companion object {
const val CHANNEL_ID = "incoming_listener"
const val NOTIF_ID = 2001
}
private var firestoreListener: ListenerRegistration? = null
private val processedCalls = mutableSetOf<String>()
override fun onCreate() {
super.onCreate()
createChannel()
startForeground(NOTIF_ID, buildNotif())
startListening()
}
private fun startListening() {
val uid = FirebaseAuth.getInstance().currentUser?.uid ?: return
firestoreListener = FirebaseFirestore.getInstance()
.collection("calls")
.whereEqualTo("receiverUid", uid)
.whereEqualTo("status", "ringing")
.addSnapshotListener { snap, _ ->
snap?.documentChanges?.forEach { change ->
val callId = change.document.getString("callId") ?: return@forEach
if (processedCalls.contains(callId)) return@forEach
processedCalls.add(callId)
val callerUid = change.document.getString("callerUid") ?: return@forEach
val callerNum = change.document.getString("callerNumber") ?: "Desconocido"
val offer = change.document.getString("offer") ?: return@forEach
if (offer.isNotBlank()) showIncomingCall(callId, callerUid, callerNum, offer)
}
}
}
private fun showIncomingCall(callId:String, callerUid:String, callerNum:String, offer:String) {
val intent = Intent(this, CallActivity::class.java).apply {
flags = Intent.FLAG_ACTIVITY_NEW_TASK or Intent.FLAG_ACTIVITY_CLEAR_TOP
putExtra("CALL_ID", callId)
putExtra("IS_CALLER", false)
putExtra("REMOTE_UID", callerUid)
putExtra("REMOTE_NAME", callerNum)
putExtra("REMOTE_SDP", offer)
}
val pi = PendingIntent.getActivity(this, callId.hashCode(), intent,
PendingIntent.FLAG_UPDATE_CURRENT or PendingIntent.FLAG_IMMUTABLE)
val notif = NotificationCompat.Builder(this, "calls")
.setSmallIcon(android.R.drawable.ic_menu_call)
.setContentTitle("Llamada entrante")
.setContentText("De: $callerNum")
.setPriority(NotificationCompat.PRIORITY_MAX)
.setCategory(NotificationCompat.CATEGORY_CALL)
.setFullScreenIntent(pi, true)
.setAutoCancel(true).setOngoing(true).build()
(getSystemService(NOTIFICATION_SERVICE) as NotificationManager)
.notify(callId.hashCode(), notif)
// Iniciar ringtone
startService(Intent(this, CallForegroundService::class.java).apply {
putExtra(CallForegroundService.EXTRA_REMOTE_NAME, "De: $callerNum") })
}
override fun onDestroy() { super.onDestroy(); firestoreListener?.remove() }
override fun onBind(i: Intent?): IBinder? = null
private fun buildNotif() = NotificationCompat.Builder(this, CHANNEL_ID)
.setSmallIcon(android.R.drawable.ic_menu_call)
.setContentTitle("SimpleCallApp activo")
.setContentText("Esperando llamadas entrantes")
.setPriority(NotificationCompat.PRIORITY_MIN)
.setOngoing(true).build()
private fun createChannel() {
if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.O) {
val nm = getSystemService(NOTIFICATION_SERVICE) as NotificationManager
nm.createNotificationChannel(NotificationChannel(
CHANNEL_ID, "SimpleCallApp", NotificationManager.IMPORTANCE_LOW))
}
}
}

3. Auditoría del Sistema de Mensajes
■ BUG 8 — listenConversation() puede emitir listas desordenadas
Problem
a:
ChatRepository usa dos listeners en paralelo (s1 para mensajes enviados, s2 para recibidos). Cuando ambos
snapshots llegan casi simultáneamente (dentro de ~50ms), se hacen 2 llamadas a trySend() con listas
parcialmente actualizadas. El resultado: el RecyclerView puede mostrar brevemente mensajes en orden
incorrecto o con un parpadeo visual.
Solución
:
Usar un Mutex para sincronizar las actualizaciones al mapa compartido, o combinar los dos listeners en un solo
query con in() cuando Firestore lo permita. Solución pragmática: agregar un debounce de 50ms antes de emitir.
■ BUG 9 — sendMessage() sin validar otherUid vacío
Problem
a:
Si ChatFragment se abre con otherUid='' (por un bug de navegación o args faltantes), sendMessage() envía un
mensaje con to=''. Este mensaje nunca aparece en ninguna conversación y ocupa espacio en Firestore.
Solución
:
Validar otherUid al inicio de sendMessage() y deshabilitar el botón Enviar si está vacío.

fun listenConversation(myUid: String, otherUid: String): Flow<List<Message>> =
callbackFlow {
val merged = mutableMapOf<String, Message>()
val mutex = Mutex()
suspend fun emit() {
val sorted = mutex.withLock { merged.values.sortedBy { it.timestamp } }
trySend(sorted)
}
